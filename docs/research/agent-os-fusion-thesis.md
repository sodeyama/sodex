# 完全自作OSにおけるエージェント融合のあるべき姿

**執筆日**: 2026-03-19
**前提**: [llm-agent-os-survey.md](llm-agent-os-survey.md) の調査結果を踏まえた設計論考
**対象**: Sodex — i486向け完全独自OSカーネル

---

## 序論: 問いの本質

「LLMエージェントをOS内部に置く」とは何を意味するのか。

AIOSはPythonでOSをシミュレーションしている。MemGPTはOSのメタファーを借りているだけだ。Composable OS KernelsはLinux LKMにテンソル処理を押し込んだが、それはOSにMLを載せたのであって、OSとエージェントが融合したわけではない。

これらはすべて、**既存のOS概念にLLMを当てはめた**プロジェクトだ。

Sodexが問うべきは逆だ: **LLMエージェントの存在を前提として、OSはどう設計されるべきか**。

---

## 1. 現状の評価: ユーザランドエージェントの限界と強み

### 1.1 現在のSodexアーキテクチャ

```
カーネル                          ユーザランド
┌──────────────────┐            ┌──────────────────┐
│ プロセス管理       │            │ agent (CLI)       │
│  └ 円形ランキュー   │            │  └ agent_loop     │
│  └ RRスケジューラ   │  syscall   │  └ claude_client  │
│ メモリ管理        │ ◄────────► │  └ tool_dispatch  │
│  └ kalloc/kfree   │  INT 0x80  │  └ session        │
│ ファイルシステム   │            │  └ permissions    │
│  └ ext3           │            │                   │
│ ネットワーク       │            │ libagent          │
│  └ NE2000+uIP     │            │  └ http/tls/sse   │
│  └ ソケットAPI     │            │  └ dns            │
│ SSH/Adminサーバ   │            │  └ json           │
└──────────────────┘            └──────────────────┘
```

### 1.2 この設計の強み

**ユーザランドに置いたことは正しい判断だった。** 理由:

1. **障害隔離**: エージェントがクラッシュしてもカーネルは死なない。API呼び出しがタイムアウトしてもOSは動き続ける。これは「OSにLLMを埋め込む」アプローチでは得られない
2. **開発速度**: カーネル再ビルドなしにエージェントの挙動を変更できる。system_prompt.txtの編集だけで済む
3. **メモリ保護**: エージェントのバグがカーネルヒープを壊さない。palloc/pfreeによるプロセス空間分離が効いている
4. **ネットワーク安全性**: API呼び出しの非決定的レイテンシがカーネルのタイマー割り込み（10ms）を阻害しない

### 1.3 しかし限界がある

**カーネルはエージェントの存在を知らない。**

- スケジューラはエージェントプロセスを他のプロセスと区別できない
- MAXPROCESS=3の固定制限で、エージェントが1プロセスを占有すると他に2つしかない
- fork()未実装のため、エージェントがサブタスクを並列実行できない
- パイプは片方向4KBで、複雑なツール連携には貧弱
- brk()未実装で、長時間実行エージェントがメモリを追加確保できない
- 共有メモリがなく、複数エージェント間のデータ共有がファイル経由のみ

**エージェントはOSを使っているが、OSの一部にはなっていない。**

## 2. 融合の設計原則

調査した全プロジェクトの教訓と、Sodexの独自制約を踏まえ、以下の原則を提案する。

### 原則1: エージェントはファーストクラスの実行単位

従来のOSでは「プロセス」が基本単位だ。エージェントOS融合では、**エージェント**がプロセスと同格の、あるいはプロセスより上位の実行概念になる。

```
従来OS:
  プロセス → スレッド → 命令実行

エージェントOS:
  エージェント → ターン → {推論, ツール実行(=プロセス群), メモリ操作}
```

エージェントの1ターンは以下の段階を含む:
1. **推論フェーズ**: LLM API呼び出し（ネットワークI/O、CPU待ち）
2. **行動フェーズ**: ツール実行（プロセス生成、ファイルI/O、さらなるネットワークI/O）
3. **記憶フェーズ**: コンテキスト更新、セッション保存

各フェーズでのリソース要求は劇的に異なる。推論中はCPUをほぼ使わない（API応答待ち）。行動中はプロセス生成とI/Oが集中する。記憶中はストレージI/Oが支配的。

**カーネルがこのライフサイクルを認識すべき理由**: スケジューラが「今はAPI待ちだからCPUを他に回せ」と判断できる。メモリ管理が「このエージェントは長期間推論中だからページアウトしてよい」と判断できる。

### 原則2: コンテキストウィンドウは仮想メモリ

MemGPTの最大の洞察を、Sodexでは**比喩ではなく実装**にする。

```
┌─────────────────────────────────────────┐
│  コンテキストウィンドウ (190Kトークン)      │  ← "物理メモリ"
│  ├── システムプロンプト (固定)              │  ← カーネルテキスト
│  ├── コアメモリ (更新可能)                 │  ← 常駐ページ
│  ├── 直近会話 (FIFO)                     │  ← ワーキングセット
│  └── ツール結果 (一時的)                  │  ← I/Oバッファ
├─────────────────────────────────────────┤
│  セッションファイル (.jsonl)               │  ← "スワップ領域"
│  アーカイバルストア                        │  ← "ファイルシステム"
└─────────────────────────────────────────┘
```

現在のsession.cのcompaction機構は「ページアウト」に相当する。不足しているのは「ページイン」— 過去の会話から関連ターンを検索して復元する機能だ。

**カーネルとの接点**: MemHoleアロケータの概念を借りて、コンテキストウィンドウ内のトークン領域を「コンテキストホール」として管理する。エビクションポリシーはLRU-K（AIOSのMemory Manager参考）。ただしベクトルDBはi486では非現実的なので、キーワード頻度ベースの軽量検索で代替する。

### 原則3: syscallの拡張ではなく、新しいsyscallクラスの創設

AIOSのsyscall分類（LLM/Memory/Storage/Tool/Access）は正しい方向だが、Pythonのシミュレーションに留まっている。Sodexでは**実際のINT 0x80に新しいsyscallクラスを追加する**。

```
既存syscall:
  1-50:   プロセス管理 (exit, fork, waitpid, execve, signal, kill)
  3-6:    ファイルI/O (read, write, open, close)
  39-42:  ディレクトリ (mkdir, rmdir, pipe, dup)
  100-123: デバイスI/O (termios, console, framebuffer)
  300:    デバッグ (memdump)
  401-413: ネットワーク (socket, bind, listen, accept, connect, send, recv)

新規syscall（提案）:
  500-599: エージェント制御
    500: agent_create     ← エージェントプロセスの生成（専用メモリリージョン付き）
    501: agent_yield      ← 推論待ち中のCPU明け渡し（ヒント付きスリープ）
    502: agent_context    ← コンテキストメモリ操作（ページイン/アウト）
    503: agent_ipc        ← エージェント間メッセージパッシング
    504: agent_quota      ← リソース使用量の照会/設定
    505: agent_audit      ← 監査ログへの構造化イベント書き込み
```

**重要**: これは「エージェント専用の特権API」ではない。通常のプロセスがこれらのsyscallを呼んでも意味がないだけだ。カーネルはtask_structのフラグでエージェントプロセスかどうかを判定する。

### 原則4: ガバナンスはカーネルの責務

OS for Agentsの「Gavel」モジュールの考え方: **エージェントの出力検証はユーザランドではなくカーネルが行うべき**。

理由: ユーザランドのpermissions.cはエージェント自身のプロセス空間で動いている。悪意あるツール実行や暴走したエージェントが権限チェックをバイパスする可能性がある。カーネル内の監視は、プロセスの特権レベル分離（Ring 0 vs Ring 3）により本質的に安全。

```
現在:
  agent_loop → permission_check → tool_execute   (すべてRing 3)

提案:
  agent_loop → syscall(tool_execute) → [カーネル: 権限検証] → 実行   (検証はRing 0)
```

これはLinuxのseccomp-BPFに近い概念だが、Sodexでは「エージェントBPF」として独自設計できる。syscallフィルタリングルールをカーネル内に持ち、エージェントプロセスからのsyscallを個別に許可/拒否する。

### 原則5: エージェントの自律性は段階的に付与

OS for Agentsの「Human-in-the-loop sudo」の発展形。エージェントの権限はランレベルのように段階的に管理する:

```
Level 0: 観察のみ（read_file, list_dir, get_system_info）
Level 1: 限定書き込み（/home/user/配下のみwrite_file）
Level 2: コマンド実行（制限付きrun_command — deny_cmdで制御）
Level 3: システム操作（manage_process, write_file全域）
Level 4: カーネル操作（将来: カーネルパラメータ変更 — OS-R1的な自己最適化）
```

現在のpermissions.confのSTRICT/STANDARD/PERMISSIVEモードはLevel 0-2に相当する。これをカーネルのcapability bitとして実装し、プロセスレベルで強制する。

## 3. Sodexだからこそ可能なこと

### 3.1 「全部自分で書いている」強み

AIOSがLinux上のPythonプロセスである限り、OSとエージェントの融合には本質的な限界がある。Linuxのスケジューラやメモリ管理を変更するには巨大なカーネルツリーを理解し、コミュニティの合意を得る必要がある。

Sodexはすべてが自前だ:

- **スケジューラ**: process.cのschedule()を直接変更できる。エージェントプロセスの状態に応じた動的優先度は数十行の変更で実現可能
- **メモリ管理**: memory.cのkalloc/kfreeに「エージェントリージョン」を追加できる。パーティション化されたメモリプールで、エージェントの暴走がカーネルヒープを侵食しない
- **syscallテーブル**: syscall.cにエントリを追加するだけ。ABIの後方互換性を気にする必要がない
- **ネットワークスタック**: NE2000ドライバからuIP、ソケット層まですべて自前。API呼び出しの優先度制御やトラフィック分類をNICドライバレベルで実装できる

### 3.2 ハードウェアからLLMまでの完全な垂直統合

```
NE2000 NIC → DMAバッファ → uIP TCP/IP → ソケット層 → DNS → HTTP/1.1 → TLS (BearSSL)
  → SSE Parser → Claude Adapter → Agent Loop → Tool Dispatch → syscall → カーネル
```

この垂直統合は他のどのプロジェクトにもない。ネットワークパケットの受信からLLMの応答処理まで、すべての層でエージェント最適化が可能だ。例えば:

- **NICドライバ**: SSEデータのためのDMAバッファ優先度を上げる
- **uIPスタック**: API通信用ソケットのTCPウィンドウサイズを最適化
- **ソケット層**: agent_yield syscallと連携し、recv待ちでプロセスを効率的にスリープ

### 3.3 カーネルサーバとエージェントの共存

Sodexは既にカーネル内にSSHサーバ、Adminサーバ、HTTPサーバを持っている。これらは**カーネルスレッド的な存在**で、ユーザランドプロセスとは別の実行コンテキストで動く。

この構造を活かし、**エージェント監視デーモン（agent supervisor）をカーネル内に配置**する:

```
カーネル空間:
  agent_supervisor
    ├── エージェントプロセスの生存監視（watchdog）
    ├── リソースクォータ強制（トークン数、API呼び出し回数、CPU時間）
    ├── 監査ログの集約（admin_serverと統合）
    └── 緊急停止（エージェントがLevel超過操作を試みた場合）

ユーザ空間:
  agent (libagent)
    ├── agent_loop（推論・行動・記憶サイクル）
    ├── tool_dispatch（ツール実行）
    └── session（セッション永続化）
```

この分離は「信頼できない計算（エージェントの推論とツール実行）はユーザランドで、信頼できる検証（権限、クォータ、監査）はカーネルで」という原則に基づく。

## 4. 具体的な融合アーキテクチャ

### 4.1 タスク構造体の拡張

```c
struct task_struct {
    // === 既存フィールド ===
    pid_t pid;
    struct hard_context *context;
    struct files_struct *files;
    struct dlist_set run_list;
    u_int32_t signal;
    sighandler_t sigactions[MAX_SIGNALS];
    u_int32_t *pg_dir;
    u_int32_t esp0;
    int state;

    // === エージェント拡張 ===
    u_int32_t agent_flags;           // AGENT_FLAG_NONE / AGENT_FLAG_ACTIVE
    u_int32_t agent_level;           // 権限レベル (0-4)
    u_int32_t agent_phase;           // PHASE_IDLE / REASONING / ACTING / MEMORIZING
    u_int32_t agent_tokens_used;     // 累積トークン数
    u_int32_t agent_tokens_limit;    // トークン上限
    u_int32_t agent_api_calls;       // API呼び出し回数
    u_int32_t agent_mem_region;      // 専用メモリリージョンのベース
    u_int32_t agent_mem_limit;       // 専用メモリリージョンのサイズ上限
};
```

agent_flags == 0ならば通常プロセスとして既存のスケジューリングパスをそのまま通る。カーネルの変更は最小限。

### 4.2 スケジューラの拡張

```c
/* schedule() の拡張 — process.c */
void schedule(void) {
    struct task_struct *next = pick_next_task();

    /* エージェントフェーズに応じた優先度調整 */
    if (next->agent_flags & AGENT_FLAG_ACTIVE) {
        switch (next->agent_phase) {
        case AGENT_PHASE_REASONING:
            /* API応答待ち — 低優先度、他のプロセスにCPUを譲る */
            /* ただしソケットにデータが来たら即座にwakeup */
            if (!socket_has_data(next)) {
                next = pick_next_non_agent_task();
            }
            break;
        case AGENT_PHASE_ACTING:
            /* ツール実行中 — 通常優先度 */
            break;
        case AGENT_PHASE_MEMORIZING:
            /* セッション保存中 — 低優先度だがI/O完了まで実行 */
            break;
        }
    }

    switch_to(next);
}
```

### 4.3 エージェント仮想コンテキスト（MemGPT実OS実装）

```
/var/agent/
├── sessions/
│   └── <uuid>.jsonl            ← スワップ領域（全会話履歴）
├── archival/
│   └── <workspace-hash>/
│       ├── knowledge.jsonl     ← 永続知識ベース
│       └── index.dat           ← キーワードインデックス
├── core/
│   └── <agent-pid>/
│       ├── user_profile.txt    ← コアメモリ: ユーザー情報
│       ├── workspace.txt       ← コアメモリ: 作業状態
│       └── facts.txt           ← コアメモリ: 学習した事実
└── shared/
    └── <shared-key>.dat        ← エージェント間共有データ
```

**コアメモリ**: エージェントが明示的にツール呼び出しで更新する固定サイズバッファ（各2KB）。コンテキストウィンドウに常に含まれる「常駐ページ」。プロセスのアドレス空間とは別に、カーネルが管理するファイルとして存在。

**アーカイバル検索**: ベクトルDBの代わりにキーワード頻度マッチング。knowledge.jsonlの各エントリにキーワードタグを付与し、index.datにソート済みキーワードリストを保持。検索はバイナリサーチ + スコアリング。i486のメモリ制約内で実装可能。

### 4.4 エージェントIPC

fork()なしでのマルチエージェント通信。既存のパイプ機構を拡張する:

```c
/* 新syscall: agent_ipc */
/* メッセージベースのIPC — パイプの上位抽象化 */

struct agent_message {
    u_int32_t from_pid;          // 送信元エージェント
    u_int32_t to_pid;            // 宛先エージェント (0=ブロードキャスト)
    u_int32_t msg_type;          // MSG_TOOL_RESULT / MSG_CONTEXT_SHARE / MSG_COORDINATE
    u_int32_t payload_len;       // ペイロード長
    char payload[AGENT_MSG_MAX]; // メッセージ本文（JSON）
};
```

カーネル内にメッセージキュー（リングバッファ）を持ち、エージェントプロセス間のメッセージングを仲介。ソケットよりも軽量で、ファイルI/Oの非効率性を避ける。

## 5. 結論: あるべき姿

### 5.1 基本姿勢

**エージェントの知性はユーザランドに、エージェントの安全はカーネルに。**

推論ループ、ツール実行、会話管理、セッション永続化はすべてユーザランドに留める。カーネルは以下だけを担う:

1. エージェントプロセスの**認識**（task_structにフラグ）
2. エージェントの**保護**（リソースクォータ、権限強制）
3. エージェントの**効率化**（フェーズ対応スケジューリング）
4. エージェント間の**通信**（カーネルIPC）

これはマイクロカーネル的な発想だ。カーネルは「メカニズム」を提供し、「ポリシー」はユーザランド（libagent）に委ねる。

### 5.2 Sodexの独自ポジション

既存のLLM-OS融合プロジェクトの問題は、**OSを書いていない**ことだ:

- AIOSはPythonでOSを**演じている**
- MemGPTはOSのメタファーを**借りている**
- Composable OS KernelsはLinuxの上にMLを**載せている**
- AgentOS / OS for Agentsはランタイムを**名乗っている**

Sodexだけが「NICドライバからsyscallテーブルまで自分で書いたOS上で、LLMエージェントが自律動作する」という構成を持つ。これは研究としても工学としても、他に類例がない。

### 5.3 目指すべき最終形

```
┌─────────────────────────────────────────────────┐
│  Claude API (外部)                                │
└────────────────────────┬────────────────────────┘
                         │ HTTPS (TLS 1.2, BearSSL)
┌────────────────────────┴────────────────────────┐
│  ユーザランド                                      │
│                                                   │
│  ┌─ Agent A (分析) ──┐  ┌─ Agent B (実行) ──┐    │
│  │ agent_loop        │  │ agent_loop        │    │
│  │ context_manager   │  │ context_manager   │    │
│  │ tool_dispatch     │  │ tool_dispatch     │    │
│  └───────┬───────────┘  └───────┬───────────┘    │
│          │   agent_ipc(503)     │                 │
│          └──────────┬───────────┘                 │
│                     │                             │
│  ┌─ libagent ───────┴──────────────────────────┐  │
│  │ claude_client  session  archival_store       │  │
│  │ http_client    dns      tls_client           │  │
│  │ sse_parser     json     core_memory          │  │
│  └─────────────────────────────────────────────┘  │
├───────────────── INT 0x80 ────────────────────────┤
│  カーネル                                          │
│                                                   │
│  ┌─ agent_supervisor ───────────────────────────┐ │
│  │ 生存監視 / クォータ強制 / 監査ログ / 緊急停止   │ │
│  └──────────────────────────────────────────────┘ │
│  ┌─ プロセス管理 ──┐  ┌─ メモリ管理 ──────────┐  │
│  │ schedule()      │  │ kalloc/kfree          │  │
│  │ + フェーズ対応   │  │ + agent_mem_region    │  │
│  │ + 動的優先度     │  │ + クォータ強制         │  │
│  └─────────────────┘  └───────────────────────┘  │
│  ┌─ ネットワーク ──┐  ┌─ ファイルシステム ────┐  │
│  │ NE2000 + uIP    │  │ ext3                  │  │
│  │ ソケット(64)     │  │ + /var/agent 永続化   │  │
│  └─────────────────┘  └───────────────────────┘  │
│  ┌─ エージェントIPC ───────────────────────────┐  │
│  │ メッセージキュー (リングバッファ)              │ │
│  │ agent_ipc syscall (503)                      │ │
│  └──────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────┘
```

この図の本質は**二重の保護環**だ:
- 外側の環: ユーザランドのpermissions.c（ポリシー）
- 内側の環: カーネルのagent_supervisor（メカニズム）

エージェントが自分自身の権限チェックを改ざんしても、カーネルの検証は突破できない。これは仮想化やWASMサンドボックスでなく、CPUの特権リング（Ring 0/3）という**ハードウェアの信頼境界**に基づいている。

### 5.4 未来の展望

1. **自己最適化OS**: エージェントがOS-R1のように、自分が動作するカーネルのパラメータ（スケジューラのタイムスライス、メモリプールサイズ、ソケットバッファサイズ）を調整する。エージェントはLevel 4権限を持ち、agent_syscallでカーネルに変更を指示する。カーネルは変更を適用し、パフォーマンスメトリクスをフィードバックする。

2. **エージェントによるデバイスドライバ開発**: エージェントがハードウェア仕様を読み、ドライバコードを生成し、カーネルモジュールとしてロード（動的ロードが実装された場合）。Sodexの全ドライバソースはext3上にあるため、エージェントがread_fileで読んで理解し、新ドライバを生成する流れは既に可能。

3. **分散エージェントOS**: 複数のSodexインスタンスがネットワーク越しにエージェント間通信。AIOS ServerのDHT/Gossipの概念を、Sodexのソケット層上に実装。各インスタンスのエージェントが協調してタスクを解決する。
