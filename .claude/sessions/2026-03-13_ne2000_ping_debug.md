# NE2000 ping疎通デバッグ — 2026-03-13

## 目標
ホスト(macOS)からQEMUゲスト(Sodex OS, 10.0.2.15)へのping疎通を実現する。

## ネットワーク構成
- QEMU起動: `./start.sh net` (vmnet-shared, sudo必要)
- NIC: `ne2k_isa`, irq=11, iobase=0xC100, mac=52:54:00:12:34:56
- ゲストIP: 10.0.2.15, netmask: 255.255.255.0, gateway: 10.0.2.1
- uIP TCP/IPスタック使用

## 修正済みバグ

### 1. write_remote_dma: RSAR0/RSAR1バイト順が逆 (ne2000.c)
- RSAR0にhigh byte、RSAR1にlow byteを書いていた（read_remote_dmaとは逆）
- 修正: read_remote_dmaと同じ順序に統一

### 2. ne2000_send: DMAアドレスがページ番号のまま
- `write_remote_dma(SEND_ADDR, ...)` → `write_remote_dma(SEND_ADDR << 8, ...)`
- 注: 元はRSARバグ(#1)と打ち消し合っていたが、#1修正に伴い必要

### 3. init_ne2000(): CR レジスタが初期化中に何度も破壊される (最重要)
- `nic_sendenable_ctl()`, `nic_recvenable_ctl()`, `init_ringbuf()`, `set_macaddr()` がCRの全ビットを上書き
- CR_STA, CR_RD_STOP が失われ、NICがSTOP状態のまま
- IMR書き込みもPage0が保証されていなかった
- **修正**: init_ne2000()をDP8390正規シーケンスで全面書き直し:
  1. Reset → STP|RD_STOP → RST待ち
  2. DCR, RBCR, RCR, TCR(loopback), IMR=0, ISR=FF
  3. PSTART/PSTOP/BNRY/TPSR (Page0)
  4. Page1: MAC, CURR, MAR
  5. Page0: ISR=FF, IMR=ALL, TCR=normal
  6. CR = STA|RD_STOP (NIC起動)

### 4. write_remote_dma: ISR RDCクリア漏れ
- DMA前にISR_RDCをクリアする必要がある
- `out8(O_ISR, ISR_RDC)` を追加

### 5. init_sendreceive_config: RCRを0にリセット
- 受信フィルタがリセットされていた → RCR_AB に修正（init全面書き直しで解消）

## 現在の状況（未解決）

### 送信は成功している
- TSR=0x01 (PTX) — パケット送信自体は成功
- ARPリクエスト(10.0.2.2宛)をゲストから送信 → SLIRPからARP reply受信を確認

### IRQは発火している
- `IRQ count=1, last_isr=0x03` (PTX+PRX) — 割り込みは1回発火
- ISRハンドラ `i2Bh_ne2000_interrupt()` は正常に呼ばれている

### パケット受信もできている
- `NE2K-RX: bnry=0x46 curr=0x47, status=0x01 next=0x47 len=0x0044`
- ARP reply (68バイト) がNE2000の受信バッファに到着

### BNRY更新問題（現在の最大課題）
- `out8(O_BNRY, next_page=0x47)` を書いているはずだが、読み返すと0x46のまま
- 同じパケットが無限に再読み取りされる
- Page0が選択されていることは確認済み（明示的にCR_PAGE0を設定）
- **次のステップ**: BNRY書き込み後のread-backでデバッグ追加済み（未テスト）
  - `ne_serial_hex8(in8(io_base + I_BNRY))` で実際の値を確認する

### ISRポーリングでPTXが検出されない問題
- ポーリングループ(50M回)でISR=0x00のまま
- しかしTSR=PTXは立っている、IRQも発火している
- QEMUのNE2000実装特有の挙動の可能性
- IMR=0x00がPage2読み取りで返るが、実際にはIMR書き込みは成功している可能性
  （ISRイベントは発生し、IRQも発火しているため）

### PICマスクが0x00
- master=0x00, slave=0x00 (全IRQ有効)
- 期待値は master=0xF8, slave=0xF7 だが、実際にはIRQは動作している
- PIC I/O自体は正常（0xAA書き込み→0xAA読み取り確認済み）
